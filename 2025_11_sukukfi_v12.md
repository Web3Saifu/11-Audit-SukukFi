## Shared ERC-20 ShareToken Causes Cross-Vault Fungibility and Misaccounting
- Severity: High

### Targets
- mint (ShareTokenUpgradeable)
- totalSupply (ERC7575VaultUpgradeable)

### Description

The protocol uses a single ERC-20 token to represent shares across all vault instances without per-vault supply tracking. This design flaw allows vaults to mint, burn, and price shares against a global total supply, rather than an isolated per-vault ledger. As a result, shares are fully interchangeable between vaults—depositing assets into one vault yields fungible tokens that can be redeemed in another vault for its underlying assets. Additionally, NAV and asset↔share conversions become inaccurate since they rely on the global totalSupply, leading to user overpayment, underwithdrawal, and pricing arbitrage.

### Root cause

ShareTokenUpgradeable and ERC7575VaultUpgradeable rely on a single global ERC-20 supply for all vaults and never record or maintain individual vault share balances. The totalSupply() function is used for both cross-vault share redemption and asset↔share conversion calculations without any per-vault context.

### Impact

1. Unauthorized Asset Extraction: An attacker deposits into Vault A, mints shares, and redeems them in Vault B to withdraw Vault B’s assets, draining vault funds. 2. Incorrect Pricing & Arbitrage: Vault NAV calculations are based on the global token supply, causing depositors to overpay or withdrawers to underreceive assets, enabling arbitrage and leading to financial losses for users.

## Incorrect allowance spending in spendSelfAllowance
- Severity: Medium

### Targets
- spendSelfAllowance (WERC7575ShareToken)

### Description

The function spendSelfAllowance is intended to decrease the vault’s allowance granted by an owner, but calls _spendAllowance(owner, owner, shares) instead of _spendAllowance(owner, msg.sender, shares). This reduces the owner’s self-allowance rather than the vault’s allowance, leaving the actual owner→vault allowance untouched and allowing vaults to overspend indefinitely.

### Root cause

A typo in the spender argument: using owner for both parameters in _spendAllowance instead of using msg.sender as the spender.

### Impact

Vaults can withdraw more shares than permitted by the owner’s allowance, potentially draining owner holdings.

## Pause Bypass via deposit finalization
- Severity: Low

### Targets
- deposit(uint256 assets, address receiver, address controller) (ERC7575VaultUpgradeable)

### Description

The three-argument deposit function does not check the vault’s isActive flag before finalizing previously requested deposits. An attacker can request a deposit while the vault is active, then pause the vault (setting isActive to false), and still call deposit(...) to mint new shares based on the pending request—bypassing the pause mechanism.

### Root cause

Missing require($.isActive) guard in the deposit(...) implementation, unlike requestDeposit(...) which enforces the isActive flag.

### Impact

Unauthorized share issuance when the vault is paused, compromising the vault’s pause mechanism and potentially inflating user positions or draining vault assets.

## Pause Bypass in ERC7575VaultUpgradeable Functions
- Severity: Low

### Targets
- requestRedeem (ERC7575VaultUpgradeable)

### Description

Both the `deposit` and `requestRedeem` flows in `ERC7575VaultUpgradeable` fail to enforce the vault’s pause state, allowing operations that should be barred when `isActive` is false. Specifically, the three-argument `deposit` method finalizes pending deposit requests without checking `isActive`, and `requestRedeem` permits new redemption requests under the same condition. These gaps completely bypass the vault’s pause mechanism.

### Root cause

Neither `deposit(uint256,address,address)` nor `requestRedeem` includes a `require(isActive)` guard or an equivalent modifier, unlike other deposit functions that correctly enforce the vault’s paused state.

### Impact

Attackers or users can continue to mint shares from previously queued deposits and enqueue new redemption requests even when the vault is deactivated. This undermines the pause feature, risks unauthorized share inflation, disrupts withdrawal scheduling, and may lead to unexpected or stuck token states.

## Incomplete KYC Enforcement on Token Transfer Operations
- Severity: Low

### Targets
- transfer / transferFrom (WERC7575ShareToken)
- transfer, transferFrom (WERC7575ShareToken)
- transfer, transferFrom, batchTransfers (WERC7575ShareToken)

### Description

The contract’s KYC logic only validates the recipient of token transfers and entirely omits sender-side checks in single and batch transfer functions. As a result, unverified or de-verified accounts can still initiate outbound transfers—both individual and bulk—undermining the intended compliance controls.

### Root cause

The implementation applies `isKycVerified` only to the `to` parameter in `transfer` and `transferFrom`, and does not consult KYC status for `msg.sender` or `from`. Additionally, the `batchTransfers` function never performs any KYC checks on either sender or recipient.

### Impact

Non-KYC’d or revoked-KYC token holders can freely move their balances to KYC-approved addresses, including via batch operations. This effectively bypasses account freezes, enabling laundering, unauthorized distributions, and violation of regulatory or business rules.

## Missing KYC Enforcement in Allowance and Permit Functions
- Severity: Low

### Targets
- permit (WERC7575ShareToken)
- approve (WERC7575ShareToken)

### Description

The contract’s KYC enforcement is applied inconsistently across token operations: while mint, burn, transfer, and transferFrom validate KYC (for recipients), both `approve` and `permit` bypass these checks entirely. As a result, unverified or de-verified accounts can grant allowances or sign permits without any on‐chain compliance verification. Approved spenders can then use `transferFrom` to move tokens from non‐KYC’d accounts to KYC‐verified recipients, undermining the intended regulatory controls.

### Root cause

Both `approve` (ERC20) and `permit` (ERC-2612) functions invoke internal allowance updates (`super.approve` and `_approve`) without any `isKycVerified` validation on `msg.sender`, `spender`, or the signature owner. The KYC logic is only enforced at certain public entry points and is not integrated into internal approval mechanisms.

### Impact

Non-KYC’d token holders can unknowingly or maliciously grant allowances to spenders who then transfer tokens to compliant addresses, effectively bypassing all on-chain KYC restrictions. This allows laundering of tokens held by unverified accounts and violates compliance requirements.

## Unlimited Vault Draining via Misused spendSelfAllowance
- Severity: Low

### Targets
- spendSelfAllowance (WERC7575ShareToken)

### Description

The function spendSelfAllowance(address owner, uint256 shares) is intended to decrement a user’s vault allowance (owner→vault) before burning or redeeming shares. However, it mistakenly calls _spendAllowance(owner, owner, shares), reducing the owner’s self-allowance rather than the owner→vault allowance. Since users can set their self-allowance to uint256.max via the ERC-2612 permit mechanism (and _spendAllowance skips decrement when allowance == uint256.max), the owner→owner allowance stays infinite and the owner→vault allowance is never consumed. As a result, any registered vault can call spendSelfAllowance repeatedly to pull an unlimited number of shares from the user without ever exhausting its allowance.

### Root cause

A hardcoded second argument of owner is passed to _spendAllowance instead of using msg.sender (the vault), so the wrong allowance mapping is decremented.

### Impact

Registered vaults can drain all of a user’s token shares without limit, bypassing intended allowance caps and enabling full asset theft.

As a result, critical checks—such as pausability, KYC verification, fees, and any other side-effects implemented in those paths—are completely bypassed, allowing the onlyValidator role to move tokens arbitrarily without enforcement of compliance logic.

### Root cause

Direct manipulation of the _balances state and manual emission of Transfer events in batchTransfers, rather than routing through the existing _update or public transfer functions that enforce compliance checks and hooks.

### Impact

A malicious or compromised validator could transfer tokens between arbitrary accounts without triggering pausable checks, KYC requirements, fee deductions, or other business-critical side-effects, potentially violating regulatory compliance and enabling unauthorized token transfers.

## Improper Vault Deregistration Allows Freezing or Orphaning of User Assets
- Severity: Low

### Targets
- burn (ShareTokenUpgradeable)
- unregisterVault (ShareTokenUpgradeable)
- unregisterVault (WERC7575ShareToken)

### Description

The ShareTokenUpgradeable contract’s unregisterVault function fails to comprehensively verify a vault’s state before removal. It only checks specific on-chain balances, pending deposit/redemption metrics, and active requesters, but omits validation of off-chain investments and outstanding share obligations. Additionally, deleting the vault→asset mapping revokes the vault’s permission to burn share tokens, and any revert in external calls can block execution entirely. As a result, vault deregistration can freeze user redemptions, orphan assets held in external protocols, and render issued shares permanently unredeemable.

### Root cause

unregisterVault relies on incomplete metrics (on-chain ERC20 balance, pending deposits, cancels, claimable redeems, active requesters, isActive) and unguarded external calls, but does not: 1) check VaultMetrics.totalAssets or availableForInvestment (off-chain assets), 2) ensure no outstanding share supply for that vault, or 3) handle external call failures without reverting.

### Impact

A malicious or compromised owner can unregister a vault that still holds user assets—either locked externally or represented by outstanding shares—leading to: 1) blocked burn() calls for redemptions (freezing user funds), 2) stranded assets in external investment protocols (orphaned funds), 3) permanently unredeemable shares across users (value loss), and 4) potential denial-of-service if external calls in unregisterVault revert and prevent remediation.

## Revoked KYC Permanently Locks Tokens
- Severity: Low

### Targets
- mint, burn, transfer, transferFrom (WERC7575ShareToken)

### Description

The contract enforces a KYC check on every token operation—mint, burn, transfer, and transferFrom—by consulting the mutable `isKycVerified` mapping and reverting if it’s false. If a user’s KYC status is revoked after tokens have been minted to them, any subsequent attempt to transfer, receive, or burn those tokens will revert. Because even the vault’s `burn` function checks the user’s KYC status, there is no mechanism to redeem or recover tokens once KYC is revoked.

### Root cause

A mutable `isKycVerified` mapping is used directly in each ERC-20 operation with no exception or escape hatch. Revoking a user’s KYC status after mint means no token operation can ever succeed for that address.

### Impact

Tokens held by a revoked user become permanently locked and inaccessible. The user cannot transfer or burn their tokens, nor can any vault or special function recover them, resulting in permanent loss of access to their assets.

## Enumeration Set Caps Cause Permanent View Denial-of-Service
- Severity: Low

### Targets
- getActiveRedeemRequesters (ERC7575VaultUpgradeable)
- getActiveDepositRequesters (ERC7575VaultUpgradeable)
- requestDeposit (ERC7575VaultUpgradeable)
- activeDepositRequesters lifecycle (requestDeposit, fulfillDeposits, deposit, mint, cancelDepositRequest, getActiveDepositRequesters) (ERC7575VaultUpgradeable)

### Description

Both the deposit and redeem request workflows rely on EnumerableSet collections (activeDepositRequesters and activeRedeemRequesters) to track controllers with pending operations. Each corresponding view function (getActiveDepositRequesters and getActiveRedeemRequesters) reverts once the set’s length exceeds a hard cap of 100 entries. Because the add operations (requestDeposit and requestRedeem) impose no upper bound, and removal only occurs on specific user actions (or is omitted entirely in some fulfillment paths), an attacker or even a large honest user base can push these sets beyond 100 entries. Once that threshold is crossed, the view functions will always revert, permanently denying service to any caller attempting to enumerate pending requesters.

### Root cause

Hard-coded maximum length checks (>100) on EnumerableSet-backed requester collections combined with unrestricted additions and insufficient or missing automated pruning. The view functions enforce a strict cap and revert rather than paginate, truncate, or handle overflow gracefully.

### Impact

• Permanent Denial-of-Service on critical view endpoints: Once >100 unique controllers have made deposit or redeem requests, getActiveDepositRequesters and getActiveRedeemRequesters will always revert, breaking UIs, off-chain scripts, and on-chain logic that depend on enumerating pending requesters.
• Disruption of redemption and deposit workflows: Protocol integrations, monitoring services, and automation relying on these views will fail, potentially halting deposits, redemptions, or automated settlement processes.
• Potential governance and operational risks: Abuse of this flaw can lock out large segments of users from visibility into their pending operations, undermining trust and requiring emergency contract interventions.

## Self-Transfer Reverts in SafeTokenTransfers.safeTransferFrom
- Severity: Qa

### Targets
- safeTransferFrom (SafeTokenTransfers)

### Description

When `sender` and `recipient` are the same address and `amount > 0`, the function debits and then credits the same balance. This results in no net change, but the post-condition requires `balanceAfter == balanceBefore + amount`. The check fails and triggers a `TransferAmountMismatch()` revert on any valid self-transfer.

### Root cause

The post-transfer balance check assumes the recipient’s balance always increases by `amount`, even when transferring to oneself leaves the balance unchanged, causing the equality check to fail.

### Impact

Legitimate self-transfers will always revert, resulting in a denial of service for workflows that rely on transferring tokens to the same address (e.g., resetting allowances or triggering onReceive hooks).

### Impact

An attacker can craft inputs with self-transfers and custom flags to shift bitmask positions and manipulate rBalances of other accounts, enabling unauthorized rBalance increases or decreases, potentially draining or inflating balances.

## Incorrect Cancellation of rBalance Adjustment Due to Intervening Modifications
- Severity: Qa

### Targets
- cancelrBalanceAdjustment (WERC7575ShareToken)

### Description

The `cancelrBalanceAdjustment` function attempts to revert a prior `adjustrBalance` call by subtracting or adding back an originally recorded delta from the account’s `_rBalances`. If other token operations (e.g., `rBatchTransfers`) have modified `_rBalances` in the meantime, the cancellation either over-corrects, under-corrects, or even reverts, failing to restore the original balance.

## Incompatibility with Fee-On-Transfer and Deflationary Tokens Due to Strict Balance Checks
- Severity: Qa

### Targets
- safeTransferFrom (SafeTokenTransfers)
- requestDeposit (ERC7575VaultUpgradeable)
- _deposit (WERC7575Vault)
- claimCancelDepositRequest (ERC7575VaultUpgradeable)
- redeem (ERC7575VaultUpgradeable)
- redeem (WERC7575Vault)

### Description

Several core token transfer functions (safeTransferFrom and safeTransfer) enforce a post-transfer balance equality check (`balanceAfter == balanceBefore + amount`). This assumption fails for tokens that burn or charge fees on transfer (deflationary tokens) and for rebasing tokens whose balances change independently. As a result, deposit, withdrawal, redemption, and related functions across the vault always revert or fail when interacting with these token types, making the vault incompatible with a significant class of ERC-20 assets.

### Root cause

The SafeTokenTransfers library implements strict equality checks on the recipient’s balance before and after each transfer without accounting for tokens that deduct fees or burn tokens on transfer (deflationary tokens) or whose supply changes autonomously (rebasing tokens).

### Impact

All attempts to deposit, request deposit cancellation, or redeem shares when using fee-on-transfer, burnable, or rebasing tokens will revert, causing a denial-of-service and locking users out of the vault’s functionality for these assets. This severely limits token compatibility and threatens user funds if such tokens are permitted.
